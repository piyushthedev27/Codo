# CODO Monitoring & Observability

This document details how to monitor the health and performance of the CODO backend. We utilize an in-memory `performanceMonitor.ts` combined with a `structuredLogger.ts` that integrates natively with GCP/Vercel log drains.

## 1. Structured Logging

All requests, submissions, and errors are logged using standard JSON format to `stdout` and `stderr`.

### Vercel Logs
When deployed to Vercel, navigate to the **Logs** tab of your project. Vercel automatically parses the JSON lines.

### Example Log Formats
**Request:**
```json
{
  "severity": "INFO",
  "timestamp": "2024-03-XXT...",
  "type": "request",
  "method": "POST",
  "path": "/api/submissions",
  "statusCode": 200,
  "durationMs": 345,
  "userId": "user_123"
}
```

**Submission:**
```json
{
  "severity": "INFO",
  "timestamp": "2024-03-XXT...",
  "type": "submission",
  "userId": "user_123",
  "challengeId": "two-sum",
  "language": "python",
  "verdict": "accepted",
  "executionMs": 1050
}
```

### Alerts & Severity
Any log dispatched via `logError()` is marked with severity `ERROR`. Consider setting up Log Drains in Vercel to pipe these logs into a system like Datadog, Grafana Loki, or GCP Cloud Logging to set up alert policies (e.g., PagerDuty on rate of 5xx errors > 1%).

## 2. In-Memory Performance Monitor

The CODO backend includes a custom, 0-dependency latency and cache tracking utility built into the application runtime.

### Accessing the Report
While currently managed in-memory, you can expose this via an admin-only endpoint (future phase) to output the JSON report generated by `getPerformanceReport()`.

### Metrics Tracked:
- **API Latency (p95):** Keeps a circular buffer of the last 1,000 requests per endpoint to calculate the 95th percentile response time.
- **Cache Hit Rates:** Monitors internal caching efficiency (e.g., unread notification counts, leaderboard fetching). High miss rates indicate caching parameters may need tuning.
- **Code Execution Times:** Averages the execution duration per programming language to identify if the sandbox/evaluator is degrading over time.

## 3. Firebase Performance Monitoring (Client-Side)
If the CODO frontend implements the Firebase Performance SDK, trace data will be available in the Firebase Console under **Release & Monitor > Performance**.
- Look for `Network Requests` matching `codo.vercel.app/api/*` to measure real-world user latency from the edge.
